    100% 整合可靠性

    本主題討論您可能無法連線至 Zencoder 的原因，以及如何確保可靠的整合。

    概覽

    對於大多數客戶而言，Zencoder是必需的軟件依賴項。雖然我們的目標是100％的正常運行時間，但有時您可能無法連接到Zencoder：

    當發生這種情況並且Zencoder關閉時，您的應用程序通常會從Zencoder獲得“ 503 Service Unavailable”響應，但您可能會收到其他錯誤（例如500）。如果您超出了API速率限制，則會收到“超出403速率限制”的響應。

    好消息：由於視頻編碼是一個異步過程，因此您可以構建應用程序，而不會遇到停機或與我們的可用性有關的問題。如果這樣做，最壞的情況是您的工作需要更長的時間。但是不會發生任何錯誤。我們強烈建議您這樣做。

    更明確地說，如果您關心可靠性，則應遵循此方法進行集成 -適用於Zencoder或與之集成的任何關鍵API。

    • 我們的服務可能會受到上游提供商（例如Amazon Web Services）的問題的影響
    • 我們有時需要執行需要暫時停機的系統維護
    • 你已經超過了你的API速率限制
    • 等等

    可靠的應用程序集成

    1. 包括一個次要網址作為備份，以防萬一無法上傳到您的主要位置。
    2. 如果您從Zencoder收到的響應代碼不成功-基本上不是200或201，不要讓工作失敗。響應碼503並不表示您的視頻無法處理。這僅表示Zencoder暫時不可用。
    3. 如果你得到一個連接錯誤嘗試連接到Zencoder時，請執行相同的操作。
    4. 同樣，將您的API請求包裝為超時。我們建議30秒超時； Zencoder通常會在不到一秒鐘的時間內做出響應，因此30秒通常是足夠的時間。
    5. 在這三種情況下-如果您收到的響應代碼不成功，無法連接或API請求超時- 將作業標記為“待處理”。
    6. 定期地，重新提交處於“待處理”狀態的任何作業。例如，您可以使用cron每分鐘執行一次。

    重新提交作業後，一切都將正常進行。這樣，提交失敗的工作只會使工作花費更長的時間，而不會給您的應用程序或用戶帶來麻煩。

    偽碼

    好的，所以這不是偽代碼-它是Ruby。但是Ruby很容易閱讀。

    1. 想像一個包含這些列的視頻表。（顯然，它將有更多，包括存儲Zencoder作業ID和Zencoder輸出文件ID的列。） 
      create_table :videos do |t|
t.string  :state
t.integer :lock_version
t.index   :state
end
    2. 視頻應包括具有以下狀態的狀態機：
      • 待處理（尚未提交給Zencoder）
      • 提交（當前正在提交給Zencoder）
      • 轉碼（已成功提交給Zencoder）
      • 完成（Zencoder完成轉碼，然後完成工作）
      • 失敗（Zencoder無法對視頻進行轉碼）
    3. 攝取新視頻時，將視頻保存為“提交”狀態，並觸發後台作業將視頻提交給Zencoder。 
      # got a new video!
video = Video.new(params)
video.state = "submitting"
video.save!
submit_to_zencoder(video)

      您確實應該為Submit_to_zencoder方法提供背景。在Ruby中，使用DelayedJob，如下所示：

      delay.submit_to_zencoder(video)

      但是我們會堅持submit_to_zencoder(video)方法為示例目的。

    4. 的submit_to_zencoder函數看起來像這樣。這應該在後台異步運行。 
      def submit_to_zencoder(video)
begin
response = Zencoder::Job.create(attributes, :timeout => 30_000)
if response.code == 201
video.state = "transcoding"
else
video.state = "pending"
end

video.save!

# Rescue any connection error. Our plugin abstracts these as
# Zencoder::HTTPError.
#
# If you're not using the Zencoder plugin, this includes things
# like Errno::ECONNRESET, Errno::ETIMEDOUT, Errno::ECONNREFUSED,
# Errno::EHOSTDOWN, and SocketError.

rescue Timeout::Error, Zencoder::HTTPError
video.state = "pending"
video.save!
end
end
    5. 經常（例如，每分鐘）嘗試重新提交處於“待處理”狀態的作業。 
      def resubmit_pending_jobs
Video.where(:state => "pending").find_each do |video|
begin
video.state = "submitting"
video.save!

submit_to_zencoder(video)
rescue ActiveRecord::StaleObjectError
end
end
end

      另外，通過在videos表中添加“ lock_version”列，我們引入了樂觀鎖定。這意味著，如果記錄在Video.find查詢和video.save，它不會將作業提交給Zencoder。這樣可以防止意外將作業兩次提交給Zencoder。您可以使用悲觀鎖定或數據庫鎖定或其他某種鎖定方法來完成同一件事。

      就這麼簡單...

      考慮到所有因素，這是一種非常簡單的方法，可確保Zencoder和您的應用程序之間100％集成的可靠性。這比僅僅天真地提交工作要多一些步驟。但它可以確保無論發生什麼情況-無論是偶爾的超時，Zencoder的意外停機還是計劃的維護-您的應用程序都能可靠運行。
